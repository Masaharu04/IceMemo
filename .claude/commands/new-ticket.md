# Notionチケット作成スキル

引数として渡された内容をもとに、LEGOの開発チケットデータベースにNotionチケットを作成する。

## データベース情報

- **データソースID**: `2b3e1b0e-884d-8020-bf9c-000bf19b0870`
- **デフォルトステータス**: `未着手`
- **スキーマ**: `名前`（タイトル）、`ステータス`（未着手/進行中/完了）、`担当者`

## 手順

### 1. チケット種別の判断

引数またはコンテキストからチケット種別を判断し、アイコンを決める:

| 種別 | アイコン | 判断基準 |
|------|--------|---------|
| バグ | 💥 | クラッシュ・誤動作・期待と異なる挙動 |
| 機能追加 | ✨ | 新しいUI・機能・画面の追加 |
| 改善 | 🔧 | 既存機能のUX改善・パフォーマンス向上 |
| ドキュメント | 📋 | CLAUDE.md・READMEなどのドキュメント整備 |
| リファクタ | ♻️ | コード品質改善（動作変更なし） |
| 設定・環境 | ⚙️ | CI/CD・xcconfig・ツール設定 |

### 2. タイトルの決定

- `[バグ]` などのプレフィックスは**つけない**（アイコンで代替）
- 簡潔かつ具体的に（例: `月・年の期限計算が不正確`）
- 体言止めが望ましい

### 3. 本文の構成

チケット種別に応じて以下のセクションを使い分ける:

**バグ・不具合の場合:**
```
## 概要
（何が問題か1〜2文で）

## 発生箇所
`ファイル名:行番号`

## 問題のコード
（コードブロックで示す）

## 影響
（ユーザーへの影響・データへの影響）

## 修正案
（コードブロックで修正例を示す）

## 優先度
高 / 中 / 低（理由も添える）
```

**機能追加・改善の場合:**
```
## 概要
（何を実現したいか）

## 背景・目的
（なぜ必要か）

## 実装方針
（どう実装するか）

## 完了条件
- [ ] 条件1
- [ ] 条件2
```

**ドキュメント・リファクタの場合:**
```
## 概要
（何を整備・改善するか）

## 記載内容 / 変更内容
（箇条書き）

## 完了条件
- [ ] 条件1
```

### 4. チケット作成

`notion-create-pages` ツールを使い以下のパラメータで作成する:

```
parent: { type: "data_source_id", data_source_id: "2b3e1b0e-884d-8020-bf9c-000bf19b0870" }
pages:
  - icon: <種別アイコン>
    properties:
      名前: <タイトル>
      ステータス: "未着手"
    content: <本文（Notion Markdown形式）>
```

### 5. 完了報告

作成したチケットのURLをユーザーに伝える。

## 注意点

- タイトルに `[バグ]` などのプレフィックスは**絶対につけない**
- ステータスは常に `未着手` で作成する（優先度フィールドは現DBに存在しない）
- 複数チケットを依頼された場合は**1件ずつ**作成して都度報告する
- 本文はコードや具体的なファイルパスを含め、実装者がすぐ着手できる粒度にする
